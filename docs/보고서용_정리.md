## 1. 프로젝트 한 줄

유튜브 영상·쇼츠와 네이버 블로그, 실방문 리뷰를 근거로 제주 카페를 추천하는 검색 서비스.
**모든 추천에 "왜 이 카페인지"를 실제 발화 원문 인용으로 증명한다** — 차별점은 설명가능성.

핵심 문장: **"검색엔진에 LLM을 붙인 게 아니라, LLM에게 검색 도구를 줬다."**

## 2. 데이터 파이프라인 (배치 — 재료를 만드는 공정)

```
수집 ──────────────────────────────────────────────
  유튜브: 키워드 격자 41개 + 카페 보강 18개 (raw 1,359건, 무손실 저장)
  네이버: 카페 1,064곳 전량 크롤 (블로그 100건 + 지역검색 5건씩)
  유튜브 쇼츠 댓글: 1,467개 영상 / 카카오맵 리뷰: 844곳 (별점 포함)
       ↓
정제 (LLM 배치, gpt-5-mini) ────────────────────────
  블로그 정제: 996곳 — 요약문 + 태그(통제 사전 20개, 코드 강제) + 폐업 검출 11곳
  실존 판정: 상호/유효스니펫/registry 3신호 → 유지 876 / 보류 134 / 제외 49
  지역 정본화: LLM 추측 라벨 폐기 → 주소에서 코드로 유도 (2단 계층, 880곳, 오염 24.6% 교정)
  카카오 place_id 787곳 (78%) — 이름 조인 은퇴, 영구 키 확보. 중복 122그룹 검출
       ↓
병합 (진행 중) ─────────────────────────────────────
  place_id 기준 정본 병합 → cards.json (카페별 카드: 요약·태그·근거·집계 신호)
       ↓
근거 극성 분리 (신규, 파일럿 완료) ──────────────────
  블로그·리뷰·댓글에서 "좋았던 점 / 아쉬운 점"을 원문 그대로 잘라 분류
  모든 인용은 코드가 원문 대조 검증 (파일럿 75건 중 검증 실패 0)
  폐업·존속 추측은 별도 큐로 분리 → 사람 검수 후 반영
```

**설계 원칙 (배치)**: 노이즈 정리는 전부 배치 타임에 LLM으로 끝내고, 런타임은 정제된
카드를 읽기만 한다. 식별자·숫자는 LLM을 우회해 코드가 운반한다 (환각 방지).

## 3. 검색 아키텍처 (런타임 — LLM은 양끝 2회뿐)

```
쿼리 "애월에서 노을 보기 좋은 카페"
  ↓
[LLM ①] 의도 분석: {유형: 조회|브라우즈|조건, 지역, 하드조건, 소프트조건}
  ↓
═══ 이하 전부 코드 (LLM 없음) ═══
  1. 지역 필터 (주소 유도 코드 기반)
  2. 조건 번역: 태그 정확매칭 → 태그 사전 임베딩 조회("노을 맛집"→노을) → 실패 시 "해석 불가" 명시
  3. 태그 매칭 + 결정론 정렬 (태그 충족 + 다수결 + 반응 신호)
  4. 0건이면 완화 사다리: 소프트조건 하나 제거 → 인접 지역 확장 (하드조건은 절대 완화 안 함)
═══════════════════════════
  ↓  카드 5~8장 + 번역 로그
[LLM ②] 종합: 근거 인용 추천문 생성 — 카드에 없는 사실 첨가 금지 (인용은 코드가 재검증)
```

보고서에 쓸 만한 포인트 세 가지:

1. **임베딩의 역할 전환** — 통상 RAG는 문서 전체를 임베딩해 유사도로 순위를 매기지만(점수가
   왜 그런지 설명 불가), 우리는 임베딩을 **사용자 언어→통제 어휘 번역기**로만 쓴다.
   임베딩 공간에 카페 1,064곳이 아니라 태그 사전 수십 개가 들어 있다. 카페 순위는
   태그 충족·언급 채널 수 같은 읽을 수 있는 신호로만 정해진다.
2. **정직한 실패** — 조건에 맞는 카페가 없으면 없다고 말하고, 조건을 하나 완화한 대안을
   "무엇을 양보했는지 라벨"과 함께 제시한다. 하드조건(애견동반 등)은 어떤 경우에도 완화하지 않는다.
3. **설명가능성은 기능이 아니라 데이터 구조** — 모든 단계가 자기 결정을 기록한 상태 객체
   (검색 깔때기: 전체 1,064 → 애월 87 → 노을 12 → 상위 5)가 파이프라인을 관통하고,
   그대로 사용자에게 "이렇게 찾았어요"로 노출된다.

## 4. 근거 표시 — 신뢰가 컨셉

- 카페마다 **좋았던 점 / 아쉬운 점** 두 칸, 실제 발화 원문 인용 (블로그 📝 / 카카오리뷰 🗺️ / 쇼츠 댓글 💬)
- 아쉬운 점도 숨기지 않는다. 반복되면 집계 표시 ("주차 불편 — 3건 반복")
- 모든 인용은 LLM 발췌를 **코드가 원문 부분문자열로 검증** — 지어낸 인용은 구조적으로 불가능
- 아쉬운 점이 없으면 "아직 아쉬운 점 언급 없음 (블로거 N명 기준)"으로 정직하게 표기

## 6. 실험으로 정한 것들 (방법론 어필 포인트)

- **임베딩 텍스트 3~4자 비교 실험**: 자연어 요약 > 구조화 프로필 (blog 5승) — 논쟁 대신 실측으로 결정
- **동료 시드 평가**: 지역 정합 18/20 vs 11/20 — 키워드 스팸 데이터는 임베딩에서 배제하고
  라벨층으로만 이식 ("층을 옮기면 독이 약")
- **정보/반응 이원화**: 블로그=정보(무엇인가), 쇼츠 댓글=반응(어떻게 느끼는가) — 매체 비교우위 분담
- **태그 사전 프롬프트 구속 실험**: LLM에게 사전을 지키라고 해도 샘 → 코드가 강제하고 이탈분은
  승격 후보로 보존
- **파일럿 → 전량 순서**: 모든 LLM 배치는 소표본 육안 검증 후 전량 (근거 극성 파일럿에서
  가십·반어 밈 오염 4유형을 실측해 프롬프트에 반영)

## 7. 현재 상태 / 남은 것

- ✅ 수집·정제·판정·지역 정본화·place_id·검색 3분기·근거 원문 인용·극성 파일럿
- 🔶 진행 중: 정본 병합(cards.json) + 재임베딩 + 서버 결정론화 / 태그 사전 v2 + 번역기
- ⬜ 예정: 의도 분석 LLM 승계, 완화 사다리, 종합 모듈, LangGraph 조립, 골든셋 평가(Hit@5),
  근거 극성 전량 + 프론트 두 칸 표시

## 8. 용어 사전 (보고서에서 통일할 것)

- **카드**: 카페 1곳의 정제된 프로필 (요약·태그·근거 인용·집계 신호). LLM에 도착하는 데이터 계약
- **다수결**: 언급 총량이 아니라 고유 채널/블로거 수 — 도배·스팸에 강한 신호
- **하드조건/소프트조건**: 가부 판정형(애견동반) / 정도형(감성). 하드는 완화 불가
- **번역 로그(TraceState)**: 쿼리 해석→필터→완화의 전 과정 기록. 설명가능성의 실체
- **정직한 실패**: 0건을 0건이라 말하고, 양보 내역을 라벨로 붙인 대안을 주는 것
