# 제주 카페 RAG — 07/08 작업 정리

> 발표 재료. 사실 + 실측 숫자 + 방법 + 발표 포인트. (0707 정리의 후속편)
> 숫자는 전부 실제 실행 로그와 데이터 파일에서 확인한 값.

## 오늘의 이야기 (회고)

처음엔 유튜브 쇼츠를 메인 데이터로 쓰려고 했다. 쇼츠 데이터셋을 만들고 실제로 임베딩 벡터까지
만들어서 검색을 돌려본 뒤의 판단이었다. 결과가 아쉬웠을 때도 데이터를 의심하기보다는
"다른 시스템으로 보완하자"는 쪽으로 생각이 갔다 — 질문을 다듬는 단계를 넣거나, 검색 로직을
더 얹는 식으로.

돌이켜보면 답은 반대쪽에 있었다. 쇼츠는 정보가 대사와 영상에 들어 있어서 API로 받아도
텍스트가 얼마 안 나온다. 그 빈자리를 시스템으로 메울 게 아니라, 텍스트가 원래 풍부한
블로그로 데이터셋을 만들어서 임베딩 정확도 자체를 올리는 게 맞았다. 실제로 그렇게 하니
같은 검색기가 다른 물건이 됐다.

그리고 예상 밖의 소득이 하나 더 있었다. 쇼츠 댓글을 크롤링해보니, 유튜버들이 영상 설명란이
아니라 댓글에 카페 상세 정보를 적고 있었다 — 주소를 📍고정댓글로 박아두고, "여기 어디예요?"라는
질문에 답을 달아주는 식으로. 사실 유튜브를 써본 사람이면 아는 사실인데, 설계할 때는 그 지식이
호출되지 않았다. 말 그대로 일종의 도메인 지식이었던 것이다.

오늘의 교훈을 한 문장으로 접으면: **데이터가 어디에 사는지에 대한 가정을, 실측이 교정했다.**
깊은 텍스트는 유튜브가 아니라 블로그에 살았고, 쇼츠의 텍스트는 설명란이 아니라 댓글에 살았다.

## 오늘 한 줄

**"모델이 아니라 데이터를 고쳤더니 검색이 좋아졌다"** — 네이버 텍스트 보강으로 low 카페 667곳을 구제하고,
웹→임베딩→검색→화면 MVP를 관통시키고, 3자 비교 실험으로 서빙용 검색 데이터를 결정한 날.

## ① 네이버 크롤링 — 텍스트 보강의 원료

**문제 인식** — 유튜브 기준 정보밀도 low가 고유 카페 1,064곳 중 667곳. 카페를 더 모아도 같은 비율로 low가 쌓임.
**가설** — low는 "세상에 안 알려진 카페"가 아니라 "유튜브 설명란이 빈약한 카페"다. 블로그엔 텍스트가 있을 것.

**방법**
- 네이버 검색 API (검색 API는 심사 없음, 일 25,000콜): 카페당 블로그 검색 100건 + 지역검색 5건
- PoC 4곳 먼저 → low 카페도 블로그 수백~수천 건 확인 → 전량 발사 (관통 먼저 원칙)
- JSONL 줄 단위 저장 (중단 안전) + 재실행 이어달리기

**결과** — 1,064곳 전량, 실패 0. **low 카페의 블로그 중앙값 1,797건 = high(1,784건)와 사실상 동일.** 가설 확정.

**발표 포인트**: "정보가 없는 카페"는 없었다. 정보가 없는 *소스*가 있었을 뿐.

## ② 재검색 구제 — 이름 정리의 힘

실존의심 85곳(블로그<5건 & 지역검색 0건)을 괄호 병기·수식어 제거 후 재검색 → **48곳(56%) 신호 회생**.
"카페보롬왓 (Cafe Boromwat)" 같은 표기 오염이 죽은 카페처럼 보이게 만들었던 것.

**발표 포인트**: 데이터가 죽은 게 아니라 검색어가 죽었던 것 — 버리기 전에 싼 구제 시도 한 번.

## ③ 판정 체계 — 3신호 분류 (살릴 것과 버릴 것)

상호(지역검색) / 유효스니펫(카페명 포함+2024이후) / registry(공공명부) 3신호 조합:
- **유지 876 / 보류 134 / 제외 49** (1,064 → 작업표 1,015)
- 단일 신호로 잘랐으면 어드브레드(스니펫 99+registry)·아베베 같은 진짜를 죽일 뻔함
- 한계도 확인: 한/영 병기 정규화 미비로 "프릳츠 제주성산점 Fritz Jeju..." 오검출 — 이름 조인의 근본 한계, place_id 필요성 재확인

## ④ 블로그 정제 (gpt-5-mini) — 오늘의 헤드라인

**방법** — Pass 1 관례 계승: json_object 강제, reasoning_effort=minimal, 슬롯 기반 요약(지어내기 금지),
카페당 유효스니펫 최신순 30개 입력, 996콜 / ~40분 / ~$2

**결과**
- **정보 부족 카페의 대량 구제 (richness 반전): low 667 → high 947 / mid 29 / low 20** — 검색에 쓸 수 있는 카페가 ~400곳에서 ~980곳으로
- **closed_hint 폐업 자동 검출 11곳** — 수리코("2026-03-02 영업 종료, 부산 이전"까지 잡음). 블로그 열화 걱정을 기능으로 뒤집음
- 카테고리 교정: 소품샵·음식점겸업 식별 (무드제주=소품샵 등)
- **태그 사전 이탈 실측**: 프롬프트 구속만으론 "밥당라떼" 같은 표기 흔들림 발생 → 코드 필터 + 이탈분 tags_extra 보존 (사전 승격 후보: 포토존 22, 핸드드립 11, 한라산뷰 10)

**발표 포인트**: LLM 비용 $2로 검색 가능한 카페가 2.4배. 프롬프트는 못 믿고 코드가 강제한다.

## ⑤ MVP 관통 — 입력→임베딩→검색→화면

- `pipeline/embed.py`: 정제본 → text-embedding-3-large → Chroma (source 라벨로 출처 분리 유지)
- `app/server.py`: FastAPI `/search` — 질문 임베딩 → top-k + 지역 필터(부족 시 완화) → 카페 단위 병합 → 카드 JSON
- `web/index.html`: 검색만 fetch로 교체, **서버 꺼지면 mock 자동 폴백** (기존 데모 무손상)
- `run_local.bat` 더블클릭 한 번으로 API(8000)+웹(8503) 기동
- UI 근거 이원화: 영상 카페=MV 트랙(썸네일), 블로그 카페=음원 트랙(그라데이션+📝후기 뱃지). 가짜 썸네일 금지

## ⑥ 비교 실험 — 논쟁을 끝낸 8문항

같은 질문 8개를 유튜브/블로그/팀원seed(구조화 프로필)/hybrid(팀원 뼈대+우리 텍스트)에 던져 top-5 비교:

| 발견 | 근거 |
|---|---|
| **블로그 5승** | "성산 오션뷰": 유튜브 top-5에 성산 0곳, 블로그 5곳 전부 성산 |
| **자연어 문장 > 구조화 프로필** | seed(카페명/태그/주소 통짜)는 점수대 전반 하락 — 임베딩 텍스트는 문장이어야 |
| **만능 요약 자석 발견** | 두루뭉술한 요약 1건이 8문항 중 4개 상위 노출 — 요약 품질 관리 지표 필요 |
| **병합 필요성 실증** | "청수당 애월점"과 "청수당 애월점 (Cheongsudang Awwol)"이 따로 검색됨 |

**서빙용 검색 데이터 확정**: hybrid 385(팀원 검증 풀 + 우리 텍스트) + blog 851(판정 유지분). 유튜브 문서는 은퇴 —
유튜브의 직업은 검색이 아니라 **근거 영상 링크·썸네일·차트인 신호**.

**발표 포인트**: "어느 데이터가 좋은가"를 회의가 아니라 실험 3번으로 결정했다.

## ⑦ 쇼츠 댓글 — 정보/반응 이원화

**프레임** — 블로그 = 정보(그곳이 무엇인가), 쇼츠 댓글 = **반응**(사람들이 어떻게 느끼는가). 매체의 비교우위.
반응 텍스트는 임베딩 금지(만능 자석 양산 위험) — 자리는 카드 카피·정렬·화제성.

**방법** — 쇼츠(≤65초)만 수집 (긴 영상 댓글은 영상 자체에 수렴, 카페 신호 없음 — 표본으로 확인).
영상당 1콜 상위 100개. 1,467편 크롤 → 댓글 보유 779편(53%), 총 14,024개.
정제(gpt-5-mini): A귀속(단일 카페 196) / B발굴(카페 미추출 559) 트랙 분기.

**결과 (755편 정제)**
- **B발굴: 카페 특정 74편** — Pass 1이 못 뽑던 쇼츠에서 댓글이 상호명을 알려줌 (📍고정댓글의 힘)
- **주소 105건** 신규 확보 / **로컬 팁 529개**(주변 탐색 재료) / **폐업 제보 10건**(성창힐 = 기존 의심 명단과 교차 확인)
- 반응 요약 586편 + 톤 분포 (긍정 288 / 혼합 181 / 부정 63)
- 바이럴 무관 영상 387편 자동 차단

**발표 포인트**: 댓글은 잡음 반, 금맥 반 — 분류기가 갈라주면 지도 정보·신규 발굴까지 나온다.

## 오늘의 인사이트 (발표 슬라이드 후보)

1. **검색 품질은 모델이 아니라 데이터에서 나온다** — 같은 임베딩 모델, 같은 질문에서 텍스트 소스만 바꿨는데 승패가 갈렸다
2. **임베딩 텍스트는 자연어 문장이어야 한다** — 구조화 프로필(태그·주소 통짜)은 유사도를 희석시킨다 (실측)
3. **신호는 이원화하라, 반복되는 패턴** — 영상 수/채널 수(유튜브) = 총량/블로거 수(블로그) = 좋아요/작성자 다양성(댓글). 총량은 오염되고, 교차가 진짜 다수결
4. **매체마다 직업이 다르다** — 블로그=정보, 유튜브 영상=근거, 쇼츠 댓글=반응, 공공명부=실존. 한 소스에 다 시키면 안 된다
5. **열화(정보 노후)는 걱정이 아니라 기능이 됐다** — closed_hint(블로그) + 폐업 제보(댓글) + registry 3중 방어
6. **우리가 만드는 건 "RAG"가 아니라 좋은 데이터 위의 검색 시스템이다** — G(생성)는 근거 없는 장식이라 보류.
   임베딩의 진짜 직업은 사전에 없는 표현("혼자 멍때리기 좋은")을 받아주는 것 — 데이터가 좋아질수록 임베딩의 구역은 좁아지지만, 그 좁은 구역이 서비스의 차별점

## 오늘 밟은 함정 (재발 방지)

- 검색어의 괄호 병기·수식어 → 진짜 카페가 유령으로 보임 (재검색으로 56% 회생)
- 프롬프트의 태그 사전 구속은 샌다 → 강제는 코드가, 이탈은 버리지 말고 보존
- 통파일 덮어쓰기 저장은 중단에 취약 → JSONL append + 이어달리기가 정답
- "이미 있으니 스킵" 로직이 절반짜리 임베딩 컬렉션을 숨김 → 소스별 문서 수를 세어보고 발견 (숫자 세기의 힘)
- 좋아요 수 ≠ 가치 — 댓글 좋아요 상위는 죄다 카페 무관 바이럴이었다
