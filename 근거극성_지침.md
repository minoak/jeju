# 근거 극성 분리 — 데이터 계약 + 표시 스펙 (7/9 확정)

> 목적: /evidence의 근거 인용을 **좋았던 점 / 아쉬운 점**으로 확실히 나눠 보여준다.
> 배치: `pipeline/evidence_polarity.py` (신규) / 통합: CP-A(정본 병합 커밋) 이후.
> FRONTEND_지침.md에 파도로 편입할 초안 — 편입 전까지 이 문서가 계약.

## 확정 결정 (7/9 커맨드 센터)

1. **혼합 문장은 절 스팬 분리 (B안)** — "뷰는 좋은데 주차는 헬" → pro/뷰 + con/주차.
   스팬은 원문의 연속 부분문자열이어야 하며 코드가 검증(결정 26 연장). 실패분 폐기.
   프론트엔 "원문 보기" 펼침 필수 (인용 왜곡 방지).
2. **aspect = 태그 사전 어휘 통일** (+ 맛/가격/서비스/청결) — 극성 데이터가 태그 신뢰도
   감사 도구를 겸함 (`근거극성_집계.json`의 `tag_audit`: 태그는 있는데 con 증거 있는 카페).
3. **카카오 별점 프리셋**: star≤2→con, star≥4→pro. LLM 판정과 충돌 시 별점 우선(로그 남김).
4. **댓글 보수 귀속**: 단일 카페 영상이거나 댓글 본문에 카페명 명시된 경우만. 애매하면 폐기.
   con 오귀속은 무고한 카페를 때린다 — 신뢰가 컨셉이므로 버리는 쪽이 정답.
   폐기된 반응은 기존 영상 단위 reaction_tone 요약으로만 유지.
5. **정직한 빈 칸**: 아쉬운 점 0건이면 숨기지 말고 "아직 아쉬운 점 언급 없음 (블로거 N명 기준)".
6. **폐업·존속 추측은 evidence가 아니다** (7/9 파일럿 실측): "일년 뒤 어찌될지" 류는 극성 칸에서
   제외하고 `폐업의심큐.jsonl`로 분리 라우팅 → 사람 검수 후 하드코딩(폐업제보 레이어).
   런타임 에이전트 서치로 실증하는 방안은 기각 — 배치 정적 데이터 원칙 유지.
7. **가십·밈·동행 불평 제외 강화** (파일럿 오염 실측 4유형 프롬프트 반영) + **aspect_raw 보존**
   — 기타(37%)의 정체를 전량 후 빈도로 측정해서 어휘 확장 여부 결정 (추측 대신 측정).
8. **인용 근거의 소스 우선순위 = 카카오리뷰 > 댓글 > 블로그** (7/9 실측: 블로그 스니펫
   20,598건 중 86% 잘림, 21% 해시태그 도배, 완결 인용 후보 4% / 카카오리뷰는 844곳
   전문 20,328건+별점). 블로그 인용은 완결문 필터 통과분(안 잘리고 #<3)만.
   **블로그의 몫은 인용이 아니라 표수("블로거 N명 언급")와 요약** — 이 수 신호는 여전히 최강.
   카카오리뷰 없는 카페(~220곳)의 아쉬운 점 칸이 비면 정직한 빈 칸 원칙(결정 5) 적용.
   본문 크롤링(스니펫→전문)은 비용·약관 리스크로 보류 — 필요성이 실측되면 재론.

## 데이터 계약 — 근거극성.jsonl 레코드

```json
{
  "spot_name": "카페한라",
  "source": "blog | kakao | comment",
  "polarity": "pro | con | info",
  "aspect": "주차편함",
  "quote": "주차는 맞은편 공영주차장 이용",
  "full_text": "…원문 전체…",
  "ref": "블로그 link | kakao:place_id | video_id",
  "likes": 802,
  "verified": true
}
```

- `quote` ⊂ `full_text` (연속 부분문자열, 코드 검증 통과분만 존재)
- `likes`: 댓글만 의미(like_count) — 대표 인용 선정 기준
- **임베딩 금지** (결정 20 유지) — 이 데이터는 표시층 전용

## 표시 스펙 (프론트)

- 카드 상세/evidence에 두 칸: **💚 좋았던 점** / **⚠️ 아쉬운 점** (info는 "정보" 접힘 칸)
- 각 인용에 source 뱃지: 📝블로그 / 🗺️카카오리뷰(별점 표시) / 💬댓글(공감 수 표시)
- 같은 aspect의 con 반복은 집계 표기: "주차 불편 — 3건 반복" (근거극성_집계의 by_aspect)
- 인용 클릭 → 원문 전체 펼침 (full_text + ref 링크)
- 아쉬운 점 0건: "아직 아쉬운 점 언급 없음 (블로거 N명 기준)" — 빈 칸 숨김 금지
- 대표 인용 우선순위: 댓글은 likes 순, 카카오는 극단 별점(1~2/5) 우선, 블로그는 최신순

## 실행 (민옥 로컬, .env OPENAI_KEY)

```bash
python pipeline/evidence_polarity.py comments --pilot 30   # ① 수율 실증 겸 파일럿 (HANDOFF 예정 실험)
python pipeline/evidence_polarity.py blog --pilot 10       # ② 블로그 파일럿 → 품질 육안 확인
python pipeline/evidence_polarity.py kakao --pilot 10      # ③ 카카오 파일럿
python pipeline/evidence_polarity.py all                   # ④ 전량 (이어달리기 — 중단 후 재실행 안전)
python pipeline/evidence_polarity.py aggregate             # 집계만 다시
```

파일럿 육안 검증 기준: ⓐ quote가 실제 원문인가(unverified 수치 확인) ⓑ con이 진짜 con인가
ⓒ 댓글 귀속이 맞는 카페인가 ⓓ 영상 감상("힐링된다")이 새어 들어오지 않았나.

## 통합 절차 (CP-A 이후 — server.py 잠금 해제 전 착수 금지)

1. /evidence 응답에 `evidence_pro[]`, `evidence_con[]` (근거극성.jsonl에서 조회)
2. 카드 표시층에 tag_audit 반영 검토 (예: 주차편함 태그 + con 3건 → 태그 톤 다운 or 주의 병기)
3. FRONTEND_지침.md에 본 문서를 파도로 편입 (표시 스펙 절 그대로)
