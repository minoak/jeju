# 프론트 지침 — /search API 계약 v2 (클로드 코드용)

> 작성: 코워크(SilenceBreaker) 2026-07-08. 백엔드 개편에 따른 프론트 대응 안내.
> 계약 원칙: **필드는 추가만 한다(하위호환). 필드가 없으면 렌더링하지 않는다. `spot_name` 조인 키는 불변.**

## 배경 (왜 바뀌나)

"해지개"처럼 카페 이름으로 검색하면 결과에 그 카페가 아예 안 나오는 문제가 실측됨
(고유명사 조회는 임베딩의 직업이 아님). 백엔드 검색이 3층으로 개편됨:

1. **이름 매치 레이어** (배포됨) — 질의에 카페명이 있으면 그 카드를 무조건 1순위 고정
2. **카드 정본(merge.py)** (작업 중) — 5개 소스 병합, 카드 필드 확장
3. **LLM 선별+근거 설명** (예정) — top 후보를 LLM이 골라 순서 매기고 "왜 이 카페인가" 생성

## 파도 1 — 이름 매치 (지금 반영 가능)

`/search` 응답 card에 추가된 필드:

```jsonc
{
  "spot_name": "해지개",
  "score": 1.0,            // 이름 매치 카드는 1.0 고정
  "name_match": true,      // ★ 신규. 질의에 이 카페 이름이 포함되어 1순위 고정됨
  ...기존 필드 동일
}
```

지역 필드도 교정·확장됨 (2026-07-08 트랙 1 — 주소 기반 교정, 오염 24.6% 수정):

```jsonc
{
  "region": "협재",          // 표시용 (세부 우선) — 기존과 동일하게 쓰면 됨
  "region_bucket": "한림",   // ★ 신규. 읍면급 버킷
  "region_fine": "협재"      // ★ 신규. 리·동급 세부 (없으면 null)
}
```

- `name_match: true` 카드는 배열 맨 앞에 옴 (이미 정렬돼 나감 — 프론트 재정렬 금지)
- 표시 제안: "이름 일치" 배지 또는 📌 아이콘. score 게이지를 쓰고 있다면 이 카드는 유사도가 아니라 확정 매치이므로 게이지 대신 배지가 맞음
- 이름 변형 중복(예: "해지개"와 "해지개카페")이 둘 다 PIN으로 올 수 있음 — **프론트에서 중복 제거하지 말 것.** 병합(merge.py)이 해결할 데이터 문제이고, 프론트가 가리면 문제가 안 보이게 됨

## 파도 2 — 카드 필드 확장 (merge.py 완성 후)

추가 예정 필드 (확정 시 본 문서 갱신):

```jsonc
{
  "address": "제주특별자치도 제주시 애월읍 애월북서길 52",  // 도로명 우선
  "phone": "064-...",        // 없으면 필드 자체 생략 가능
  "closed_hint": false,       // true면 폐업 의심 — 카드에 경고 표시 필요
  ...
}
```

이 중 일부는 이미 배포됨 (2026-07-09 동료 시드 라벨 편입):

```jsonc
{
  "caution": ["주차", "웨이팅"],   // ★ 배포됨. 주의 신호 — 카드에 ⚠ 뱃지/문구 제안
  "hours_hint": "매일 10:00~21:00", // ★ 배포됨. 영업시간 힌트 (블로그·댓글 발 — '참고용' 표기 권장)
  "reaction_tone": "혼합",          // ★ 배포됨. 쇼츠 댓글 여론 톤: 긍정|혼합|부정|"" — 이모지 뱃지 제안
  "reaction_hint": "자리 확보가 어렵고..." // ★ 배포됨. 여론 한 줄 (부정·혼합도 그대로 노출 — 신뢰가 컨셉)
}
```

- 렌더링 규칙 동일: **없으면 안 그린다.** mock 폴백(cards.js)에도 같은 필드를 넣을 필요 없음 — 없으면 미표시로 동작해야 함
- 좌표(`lat`/`lng`)는 기존과 동일하게 카드에 실려 나감. plotCards의 keywordSearch 임시방편은 이 좌표로 교체 가능 (조인 키 = `spot_name`, 기존 합의 유지)

## 파도 3 — LLM 선별+근거 설명 (★ 배포됨 2026-07-08) + 브라우즈 모드

응답 최상위에 추가:

```jsonc
{
  "query": "...",
  "intro": "질문에 대한 1~2문장 응답",   // ★ resultIntro 자리에. 실패 시 빈 문자열
  "browse": true,                        // ★ "애월 카페"류 조건 없는 질의 = 브라우즈 모드
  "total": 108,                          // ★ 브라우즈일 때 지역 내 전체 카페 수 (아니면 null)
  "cards": [ { "reason": "카드별 추천 이유 한 줄 (없으면 필드 생략)", ... } ]
}
```

- `reason` = 카드 데이터 근거로만 생성. 카드 내 1줄 표시 제안
- **브라우즈 모드**: 카드가 12개+ 오고 순서는 유사도가 아니라 다수결(고유 블로거 수). score는 0.0 —
  score 표시 로직이 있다면 browse=true일 때 숨길 것. "애월 카페 108곳" 식으로 total 노출 제안
- **지연 실측 6~9초**: 로딩 상태(스켈레톤/스피너) 필수. mock 폴백 타임아웃 15초로 상향.
  급하면 `/search?...&explain=0`으로 LLM 층 끌 수 있음 (intro/reason 없이 1~2초)

## 파도 4 — /evidence 근거 보기 (★ 배포됨 2026-07-09)

카드에 "근거 보기" 버튼 → `GET /evidence?name={spot_name}&q={사용자 질의}`:

```jsonc
{
  "n_snippets": 97, "n_matched": 21, "bloggers_matched": 21,  // "블로거 21명이 언급"
  "opinion": "방문자들은 ... 말합니다",         // LLM 의견 요약 (원문 근거만)
  "quotes": ["대화가 거의 없이 아주 조용한..."], // 실제 블로그 원문 발췌 (코드 검증됨)
  "youtube_reactions": [ {"video_id": "...", "tone": "긍정|혼합|부정", "summary": "댓글 반응 요약"} ]
}
```

- 표시 제안: "블로거 21명이 '조용'을 언급" 헤드라인 + 인용 말풍선 + 영상반응(톤 이모지)
- 응답 2~5초 (클릭 트리거라 로딩 스피너로 충분). 첫 호출은 인덱스 로드로 +2초
- tone은 그대로 노출 — 부정·혼합도 숨기지 말 것 (신뢰가 컨셉)

## 파도 5 — 이름 검색 구분선 + 여론 뱃지 (2026-07-09 확정)

1. **"비슷한 카페" 구분선** (민옥·코워크 합의): 카드 배열에 `name_match: true`가 있으면,
   첫 non-name_match 카드 앞에 구분선/소제목 삽입 — 예: `─ 비슷한 카페 ─`.
   1위는 확정 매치, 나머지는 추천이라는 걸 화면에서 읽히게. (제거 아님 — 배열 그대로)
2. **여론 뱃지**: `reaction_tone` (긍정|혼합|부정|"") → 이모지/색 뱃지, `reaction_hint` 한 줄.
   부정·혼합 숨김 금지 (신뢰가 컨셉).
3. **filtered 표기(선택)**: 응답 최상위 `filtered: N` — LLM이 무관 판단으로 제외한 수.
   N>0이면 "관련도 낮은 결과 N개 제외" 캡션 제안.
