카페인제주 ☕ — 근거로 증명하는 제주 카페 추천 RAG

유튜브 영상·쇼츠, 네이버 블로그, 실방문 리뷰를 근거로 제주 카페를 추천하는 검색 서비스.
모든 추천에 "왜 이 카페인지"를 실제 발화 원문 인용으로 증명한다 — 차별점은 설명가능성.

핵심 테제: "검색엔진에 LLM을 붙인 게 아니라, LLM에게 검색 도구를 줬다."
노이즈 정리는 전부 배치 타임에 LLM으로 끝내고, 런타임 검색의 몸통은 전부 코드다.
식별자·숫자·인용문은 LLM을 우회해 코드가 운반한다 (환각의 구조적 차단).

2. 설계 결정 세 가지

① 임베딩의 역할 전환. 통상 RAG는 문서 전체를 임베딩해 유사도로 순위를 매긴다 —
점수가 왜 그런지 설명할 수 없다. 여기서는 임베딩을 사용자 언어 → 통제 어휘 번역기로만
쓴다. 임베딩 공간에는 카페 1,064곳이 아니라 태그 사전 수십 개가 들어 있고, 카페 순위는
태그 충족·고유 채널 수 같은 읽을 수 있는 신호로만 정해진다.

② 정직한 실패. 조건에 맞는 카페가 없으면 없다고 말한다(honest_zero). 조건을 완화한
대안은 "산방산뷰 조건을 빼고 찾았어요" 같은 양보 라벨과 함께만 제시한다.
하드조건(애견동반·노키즈존 등)은 절대 완화하지 않는다 — 주차가 필수인 사람의 신뢰 문제.

③ 설명가능성은 기능이 아니라 데이터 구조. 모든 단계가 자기 결정을 기록한 상태 객체
(TraceState)가 파이프라인을 관통한다. 검색 깔때기(전체 1,064 → 애월 87 → 노을 12 →
상위 5)가 그대로 사용자에게 "이렇게 찾았어요"로 노출된다.

7. 한계와 남은 것 (정직하게)

- 검색 품질 정량 지표 부재 — 시스템 동작 검증(시나리오 8종)은 있으나, 검색 결과의
좋음을 재는 Hit@5는 골든셋 미작성으로 미구현. 다음 단계 1순위.

- LangGraph 조립 미착수 — w234_검증.py의 순수 함수 체인이 그래프 배선의
청사진이나, 실제 그래프 조립은 하지 않았다. 현재는 결정론 체인으로 충분해 우선순위를
평가(골든셋)에 내준 판단.

- 근거 극성 전량 미실행 — 파일럿(75건, 검증 실패 0)까지. 전량 실행은 비용·시간 문제.

- 재현성 한계 — 유튜브 수집은 시점 의존이라 완전 재현 불가. 확보 데이터를 git에 포함.
